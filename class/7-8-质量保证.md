# 质量保证(下半节课)

- 异常交互: 指出现异常给出相关异常信息

- PD: 产品定义文档 (Product Definition Document)

- PRD: **产品需求文档** (Product Requirements Document)

- **主测试分** 或 **主要测试评分**: 软件测试中评估测试覆盖、质量或风险的关键指标

- CR: 代码审查

  - ```
    缩写/术语	全称/含义				说明
    LGTM	 Looks Good To Me	评审者表示代码看起来没问题，可以合并
    NACK	 Negative Acknowledgement	明确表示不接受这次代码改动
    WIP	     Work In Progress	提示此功能尚未完成，仅供初步查看
    PTAL	 Please Take A Look	请求项目Owner或特定人员查看代码
    RFC		 Request For Comment	表示认为某个想法很好，邀请大家一起讨论
    CR		Code Review	代码审查
    PR		Pull Request	拉取请求（在GitHub/GitLab等平台发起CR的常见方式）
    
    ```

- 灰度阶段: 请人体验, 监控效果

  > 灰度阶段，通常指**从新版本开始向部分用户推送，到最终全量覆盖所有用户（或决定回滚）的这段时间**。它是一种**平滑过渡的发布策略**，核心思想是“**小步快跑，逐步放量**”。

  - 内灰: 内部人员
  - 外灰: 外部人群

- 舆情处理: 

  - 联系公关

    > “公关”，指的是**公共关系**，它是一个组织为了生存发展，通过传播沟通、塑造形象、平衡利益、协调关系，从而与公众建立良好关系的管理活动

  - 联系法务

- 幂等: 指防止重复操作多次请求出现业务问题

  > **无论执行一次还是多次，对系统产生的最终影响是完全相同的**

- ```
  特性维度	离线表													实时表
  核心定义	不实时更新，定期批量同步，用于历史数据分析  				数据产生后立即或极短时间内被处理，提供近乎实时的数据 
  		
  数据时效性	T+1（通常只能看到前一天的数据） 						秒级/毫秒级延迟，极低延迟 
  
  更新方式	批处理：定时（如每日凌晨）通过ETL任务全量或增量同步 	  流式处理：持续不断地捕获数据流并处理 
  
  技术栈示例	Hadoop生态（Hive, Spark, MapReduce） 				流计算框架（Flink, Storm）、消息队列（Kafka）
  
  主要优势	高准确性（最终状态）、可处理海量历史数据、技术成熟稳定 	 低延迟、实时性强、所见即所得 
  
  主要劣势	时效性差，无法及时反映最新变化 						 资源消耗大（需持续计算）、数据周期通常较短（如保留7天） 
  
  典型应用场景	财务结算、月报/季报、历史趋势分析、数据挖掘模型训练 		 实时大屏、实时监控告警、个性化推荐、实时风控、实时库存查询 
  
  数据质量监控侧重	监控ETL任务是否成功运行、数据是否完整（记录数、字段缺失）、准确性（指标波动）、一致性（多表间口径统一）									监控数据延迟（是否在可接受范围内）、处理吞吐量、流处理任务稳定性、实时指标准确性
  
  ```

- **任何涉及资金配置的操作，必须由业务方（如财务、风控、运营）直接执行，严禁通过技术手段（如SQL初始化、DRM等）或测试人员间接操作。**

  ### 🔍 关键概念解析
  1.  **资金相关的配置操作**：
      *   指任何会直接影响资金流向、账户余额、费率、分成比例、支付渠道、退款规则等的操作。
      *   **示例**：修改用户账户余额、调整商品价格、配置支付渠道费率、设置退款规则、分配营销预算等。
  2.  **对应的业务方**：
      *   指拥有资金管理权限和专业知识的人员或部门。
      *   **示例**：财务人员、风控专员、运营经理、产品经理（在特定权限下）。
  3.  **技术或测试**：
      *   指开发人员、测试人员、运维人员等。
      *   **为什么不能让他们操作？** 因为他们通常不具备资金管理的专业背景和责任，操作失误可能导致严重资金损失，且缺乏必要的审计和审批流程。
  4.  **SQL初始化或DRM等形式**：
      *   **SQL初始化**：指通过直接执行SQL语句（如 `UPDATE`、`INSERT`）来修改数据库中的资金相关数据。这种方式**绕过了所有业务逻辑、权限校验和审计记录**，风险极高。
      *   **DRM (Data Relationship Management)**：通常指数据关系管理，但在这里可能泛指**任何绕过业务系统的直接数据修改方式**，如直接操作数据库、修改配置文件等。
      *   **为什么禁止？** 因为这些方式**不可控、不可追溯、不可审计**，一旦操作失误，可能造成灾难性后果，且事后难以追责。
