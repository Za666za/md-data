# 测分

- 测试对象
- 可能风险点

- 细分和PRD差异

- 测分是分析, 用例是落地

- 核心在补充异常分支和边界场景

- 具体要从链路视角观察, 调用链路, 状态链路, 权限链路
  - 如: 创建部门的编码重复情形, 上级不存在情形, 负责人变更对应的权限的刷新,并发更新同一个部门情形

- 测试的场景要在PRD 或 系分里面有对应

- 个人觉得功能模糊有问题的点

## 测分流程

- 收集文档
- 理解需求
  - PRD - 系分互相对应关系
  - 一个模块可能存在多个功能
- 实现测分
- 列出每次需要测的范围对应的接口
- 分等级, 系分与`prd`的差异是级高等级的风险
  - 尤其注意`PRD`变更导致的模块删除或修改
  - 或系分和`prd`规定实现细节不同
  - 列出差异最后更新

- 交互稿十分重要, 要提前定好

## 测分模版

- 项目概述
- 项目整体分析
- 术语表
- 架构分析
  - 接口
  - 系统模型
    - DB
    - 缓存
    - DRM配置
  - 系统依赖
- 功能性的测试分析
  - 模块
  - 每一个模块的流程
  - 模块的交互
- 非功能性的分析
  - 资金
  - 安全
  - 并发
  - 幂等
- 三板斧
  - 可灰度
  - 可监控
    - 异常监控
    - 怎么灰度切流
  - 可回滚
    - 开关和回滚方案





- 常用敏捷开发方式, 从前一个模块上开始迭代 , 考虑当前功能对老功能影响
- 若有组件文档(项目管理的非本应用的文档),要看依赖组件
