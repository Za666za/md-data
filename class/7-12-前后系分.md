# 前端系分

## 系分目的

- 防止返工

## 项目文档

- 用得上的文档 + 仓库地址

## 项目排期

- 通常在日历上标注

## 核心: 详细设计

- 按功能或页面拆分
- 第一项: 示意图(UED) - 贴图片
- 什么功能,什么逻辑,页面展示什么东西,字段必填选填约束,先选项,
- 若是表单,表单搜索项;
- 若是表格表格展示什么内容,若表格有操作,操作的枚举是什么
- 当前功能点和页面需要调用哪些接口,接口属于后端规范已有还是未定义的
- 未定义的最后总结方便之后补上去
- 页面逻辑处理
  - 若是表格: 逻辑简单
  - 若是其他功能繁琐的:
    - 流程图梳理, 
    - 使用什么组件,
    - 数据从哪里来
    - 怎么实现的
    - 状态流转
    - 数据传送
    - 示例:
      - 5.1.3商城首页逻辑
        1.首页初始化时[进行什么请求]并行请求point.mall.home.query +point.user.info (积分余额二次确认，避免缓存不准）
        2.列表采用无限滚动加载[还是懒加载]，滚动到底部200px时自动加载下页[还是其他处理]；itemList 返回数量< pageSize 时标记列表到底，不再请求
        3.商品卡片点击跳转至商品详情页，若stockStatus==='SoLD_OUT'点击后弹Toast"已兑完，下次早点来~"
        4.积分余额区域点击跳转至积分明细页 (站内已有页面)
        5.2.3商品详情页逻辑
        1.页面初始化时调用 point.mall.item.detail，若remainStock===0 则兑换按钮直接置灰，文案"已兑完"
        2.exchangeLimit >0 且 userExchangedCount >= exchangeLimit 时，兑换按钮置灰，文案"已达兑换上限"
        3.含现的兑换需先拉起收银台，收银台回调成功后拿到payTradeNo再调兑换接□
        4.实物商品兑换前需校验收货地址是否完整，不完整时弹地址编辑弹窗
    - 若PRD存在模块或细分在某些地方有新的疑惑总结出来留作之后探讨
- 接口规范:
  - 有接口名字
  - 入参,返回
  - 参数字段,类型,说明,是否必填及可为空
- 风险评估(存在则可写)
  - 如秒杀并发
  - 余额缓存延迟(后端接口delay )
- 性能保障(可写)
  - 如接口降级,懒加载
- 监控埋点(可选)
  - 显示页面情况
- 变更记录(可选)
  - 需求或接口变更记录
- 页面布局思路等
  - 上下三栏, 还是左右结构, 先结构拆分，还是上面信息下面三个tab各自分
  - 之后构思前端需要传入数据, 这里涉及权限管控,未授权访问处理逻辑(处理逻辑如重定向或返回403,前端服务端检验等)
  - 页面涉及接口[考虑其他情况,如用户将页面tab在浏览器新页面打开编辑并返回]
  - 根据页面方式不同如路由打开考虑是否需要后端其他接口
  - 如页面表单考虑 Ant Design 组件实现
  - 考虑页面虽然不展示但实际需要请求的字段, 如id
  - 考虑各种情况默认处理,[如: 未保存直接提交的处理,未修改直接保存的处理, 直接点击页面上取消按钮是否返回返回上一页或进行其他的处理, 新增页面编辑后没保存直接关闭应该是不会存储的等]
  - 前后端要将结构协调清楚, 如部门结构,树结构,就需要后端返回
  - 前端减少数据转换的逻辑, 要后端实现好
  - 前端中的在于页面展示 + 页面交互 + 调接口
  - 考虑PRD未提但实际场景默认需要的东西
  - 拆的时候按导航拆, 拆完导航按页面拆
- 最重要的在于: 拆模块, 排日期, 每个模块怎么实现
