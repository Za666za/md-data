这个类实现全局异常捕获的核心在于 **`@RestControllerAdvice`** 和 **`@ExceptionHandler`** 这两个注解的配合使用。
我们分步骤来拆解它的实现原理：
### 1. 核心注解：`@RestControllerAdvice`
这个注解是 Spring 提供的用于全局拦截 Controller 层异常的利器。它实际上是 `@ControllerAdvice` 和 `@ResponseBody` 的结合体。
*   **拦截范围：** 它会拦截所有进入 Controller 层的请求。如果在 Controller 方法中（或者切面中）抛出了异常，Spring 就会捕获这个异常，并交给带有 `@RestControllerAdvice` 的类来处理。
*   **自动包装响应：** 因为包含了 `@ResponseBody`，所以它处理完异常后，返回的对象（如 `ApiResponse`）会被 Spring 自动序列化为 JSON 返回给前端。
### 2. 异常匹配：`@ExceptionHandler`
这个注解用来指明“当前方法处理哪种类型的异常”。
当系统中抛出一个异常时，Spring 会在这个 `GlobalExceptionHandler` 类中寻找匹配的 `@ExceptionHandler`。
你提供的代码中定义了两个处理器，这里体现了 Spring 异常处理的**精确匹配优先原则**：
#### 处理器一：处理自定义业务异常
```java
@ExceptionHandler(BusinessException.class)
public ApiResponse<Void> handleBusinessException(BusinessException exception) {
    return ApiResponse.fail(exception.getCode(), exception.getMessage());
}
```
*   **触发时机：** 当代码中抛出 `BusinessException`（比如：`throw new BusinessException(400, "用户名不存在");`）时触发。
*   **处理逻辑：** 拿到异常对象 `exception`，提取里面的错误码和错误信息，封装到统一响应体 `ApiResponse` 中返回给前端。它不会打印错误日志，因为业务异常属于预期内的正常流程反馈。
#### 处理器二：处理所有其他未知异常（兜底）
```java
@ExceptionHandler(Exception.class)
public ApiResponse<Void> handleException(Exception exception) {
    log.error("Unhandled exception", exception);
    return ApiResponse.fail(500, "server error");
}
```
*   **触发时机：** 当代码中抛出的不是 `BusinessException`，而是其他异常（比如 `NullPointerException`、`ArrayIndexOutOfBoundsException` 等系统级异常）时触发。因为 `Exception.class` 是所有异常的父类。
*   **处理逻辑：** 这种异常通常是代码 bug 导致的，属于预期外的严重错误。所以它做两件事：
    1.  **记录日志：** `log.error("Unhandled exception", exception);` 把完整的错误堆栈打印到日志文件中，方便程序员事后排查。
    2.  **隐藏细节：** 给前端返回一个模糊的提示 `"server error"` 和状态码 `500`，避免把敏感的代码错误信息（如数据库报错）直接暴露给前端用户。
### 3. 整个执行流程（结合你之前的 AOP 代码）
假设你写了之前的参数校验切面，并且抛出了 `RuntimeException("参数不能为空")`，整个流转过程是这样的：
1.  前端请求 `/api/aop/check`（没传参数）。
2.  Spring MVC 放行请求进入 Controller 方法。
3.  **AOP 切面拦截**执行，发现参数为空，执行 `throw new RuntimeException("参数不能为空");`。
4.  此时 Controller 方法并没有正常返回，而是抛出了异常。
5.  **`@RestControllerAdvice` 介入**：Spring 捕获到了这个 `RuntimeException`。
6.  **异常匹配**：Spring 去全局异常处理器里找，发现没有专门处理 `RuntimeException` 的方法，于是向下找它的父类，匹配到了 `@ExceptionHandler(Exception.class)`。
7.  **执行兜底逻辑**：执行 `handleException` 方法，在后台打印出长长的错误堆栈日志，然后给前端返回 HTTP 200，Body 为 `{"code": 500, "msg": "server error"}`。
> **小提示：**
> 如果你想让之前 AOP 抛出的“参数不能为空”能够直接返回给前端，你可以在 `GlobalExceptionHandler` 中再加一个专门处理 `RuntimeException` 的方法，或者直接把 AOP 里的 `throw new RuntimeException(...)` 改成 `throw new BusinessException(400, "参数不能为空")`。
